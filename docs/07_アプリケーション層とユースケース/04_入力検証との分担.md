# 入力検証との分担

入力検証には種類があります。形式の検証、ユースケースの前提確認、業務ルールの検証を同じ場所に置くと、責務が見えにくくなります。

| 検証 | 置き場所の例 |
| --- | --- |
| 必須、桁数、形式 | API / DTO / Validator |
| 対象データが存在するか | Application Service |
| 注文を確定できる状態か | Domain model |
| 金額が負ではないか | Value Object |

ドメインモデルは、常に不正な状態にならないように守ります。API 側の検証は、早く分かりやすいエラーを返すための入口の検証です。

同じ検証が重複する場合もありますが、目的が違うなら許容できます。入口ではユーザー向けの説明、ドメインでは不変条件の保護を重視します。

**業務上壊れてはいけない制約は、ドメインモデル側で守る**のが基本です。
