# DDD が向いている場面

DDD は、業務ルールが複雑で、関係者の言葉を揃えることが重要なシステムに向いています。単純な CRUD や管理画面だけなら、DDD の構造が重くなることがあります。

| 向いている | 向いていないことが多い |
| --- | --- |
| 業務判断が多い | データ登録と検索が中心 |
| 用語の解釈が部署ごとに違う | 仕様がほぼ画面項目で決まる |
| 長く育てるプロダクト | 短期間の使い捨てツール |
| 変更が継続的に入る | 変更頻度が低い |

DDD を始める判断では、技術的なかっこよさよりも「業務の理解が設計のボトルネックになっているか」を見ます。

**DDD は複雑さを消す手法ではなく、複雑さを扱う場所を決める手法**です。
