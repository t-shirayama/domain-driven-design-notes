# 業務ヒアリング

DDD のヒアリングでは、画面項目よりも業務判断を聞きます。「何を入力するか」だけでなく、「なぜそう判断するか」「例外は何か」「誰が困るか」を確認します。

| 聞くこと | 目的 |
| --- | --- |
| その言葉の意味 | ユビキタス言語を作る |
| 判断基準 | ドメインルールを見つける |
| 例外ケース | 境界と不変条件を見つける |
| 手作業の理由 | 暗黙知を見つける |
| 変更が多い場所 | Core Domain を見つける |

「普通は」「だいたい」「場合による」が出たら、モデルのヒントです。そこには条件分岐や境界が隠れていることが多いです。

**業務ヒアリングは、要望収集ではなくモデル発見の場**です。
